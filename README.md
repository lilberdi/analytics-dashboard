# Analytics Dashboard

Мини-дашборд заказов с интеграцией **RetailCRM → Supabase**, Telegram-уведомлениями о крупных заказах и веб-интерфейсом на **Next.js**.  
Проект готов к локальному запуску и деплою на **Vercel**.

---

## Описание проекта

Проект решает задачу мониторинга заказов в одном потоке:

- заказы забираются из RetailCRM по API (поллинг),
- сохраняются/обновляются в Supabase (таблица `orders`),
- при крупных заказах отправляются уведомления в Telegram,
- дашборд показывает график и список заказов.

---

## Функционал

- Поллинг новых заказов из RetailCRM (по `createdAt`).
- Сохранение заказов в Supabase через `upsert`.
- Telegram-уведомления для заказов > **50 000 ₸**.
- Веб-дашборд на Next.js:
  - график сумм заказов,
  - список последних заказов,
  - обновление данных из Supabase (в т.ч. realtime-подписка).

## Инструкции по запуску

### 1) Установка зависимостей

```bash
npm install
```

### 2) Запуск бота (поллинг + уведомления)

```bash
node scripts/bot.js
```

### 3) Запуск дашборда локально

```bash
npm run dev
```

Откройте: [http://localhost:3000](http://localhost:3000)

### 4) Деплой на Vercel

1. Запушить проект в GitHub.
2. Импортировать репозиторий в Vercel.
3. Указать Environment Variables:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_KEY`
4. Для бота лучше использовать отдельный процесс/сервер (например, Railway/Render/VPS), т.к. Vercel не предназначен для постоянного `setInterval` процесса.

## 💡 Советы

- Для теста бота создайте в RetailCRM тестовый заказ > `50000`, затем проверьте лог:
  - `[supabase] Upserted ...`
  - `[telegram] Sent alert for order ...`
- Если уведомления дублируются, проверьте уникальность `id` в Supabase и что `upsert` идет с `onConflict: "id"`.
- Если дашборд пустой:
  - проверьте `NEXT_PUBLIC_SUPABASE_URL/KEY`,
  - убедитесь, что таблица `orders` заполнена,
  - проверьте RLS-политики Supabase для чтения.
- Для стабильного продакшена разделите:
  - **frontend** на Vercel,
  - **бот/поллинг** на отдельный always-on хост.
